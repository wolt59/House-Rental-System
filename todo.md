# 智能房屋租赁系统 — 问题与改进清单

> 基于代码审阅（后端 FastAPI + 前端 Vue 3），按类别整理现存问题、缺失功能及优化建议。

---

## 一、合同模块（需重构）

### 1.1 合同创建流程不合理
- **现状**：房东通过前端手动输入房源ID、租客ID创建合同；租客通过"申请签约"自动创建合同（auto-create endpoint）。
- **问题**：房东可以不经过租客同意直接创建合同，不符合真实租赁流程。
- **建议流程**：租客申请 → 房东收到通知 → 房东基于申请创建/发送合同 → 双方签署 → 生效。

### 1.2 缺少合同取消功能
- **现状**：合同只有 `terminate`（终止已生效合同）接口，无 `cancel`（取消未签署合同）接口。
- **影响**：一方创建合同后，另一方无法拒绝，创建方也无法撤回。
- **建议**：新增 `pending_sign` / `pending_landlord_sign` / `pending_tenant_sign` 状态下的取消接口，取消后改为 `cancelled` 状态。

### 1.3 缺少撤回签署功能
- **现状**：一旦 landlord 或 tenant 签署，不可撤销。
- **问题**：一方误签后无补救手段。
- **建议**：在合同生效前允许撤回自己的签署（双方未全部签署时可撤回）。

### 1.4 缺少拒绝合同功能
- **现状**：没有明确的拒绝途径，只能不操作。
- **建议**：新增拒绝接口，将合同状态设为 `rejected`，并发送消息通知对方。

### 1.5 缺失状态的枚举/常量管理
- **现状**：状态字符串（`"pending_sign"`、`"active"`、`"terminated"` 等）硬编码在整个代码中。
- **建议**：定义 Python Enum 和前端常量，避免拼写错误。

### 1.6 合同自动创建时跳过价格协商
- **现状**：`auto_create_contract` 直接使用 `property_obj.rent` 作为月租金。
- **问题**：租客无法与房东协商租金、押金等条款。
- **建议**：允许双方在签署前修改合同条款，或提供"房东起草 → 租客确认"的流程。

## 二、消息系统（建议重构为聊天式）

### 2.1 消息以表格展示，非聊天界面
- **现状**：`Messages.vue` 使用 `el-table` 展示收/发消息，留言式而非聊天式。
- **建议**：改造成类似微信/钉钉的聊天界面：
  - 左侧：会话列表（按最后消息时间排序）
  - 右侧：选中会话的消息气泡（自己发的在右侧，对方在左侧）
  - 底部：输入框 + 发送按钮

### 2.2 发送消息需手动输入接收人ID
- **现状**：需要输入数字 ID 才能发送消息，极其不友好。
- **建议**：改为从房源详情页"联系房东"自动填充接收人；聊天界面可通过搜索用户名选择聊天对象。

### 2.3 缺少会话（Conversation）概念
- **现状**：`messages` 表只有单条消息，无会话分组。
- **建议**：无需新增表，前端按 `from_user_id + to_user_id` 分组聚合，展示最近一条消息作为会话摘要。

### 2.4 缺少未读消息计数
- **现实**：侧边栏"消息中心"无未读红点/徽标。
- **建议**：新增未读总数的 API 或在 `get_received_messages` 中返回未读计数。

### 2.5 无实时消息推送
- **现状**：需手动刷新页面才能看到新消息。
- **建议**：引入 WebSocket（如 Socket.IO）实现实时消息推送和接收。

### 2.6 消息内容限制 1000 字符
- **现状**：数据库 `content` 字段为 `String(1000)`。
- **建议**：对于聊天场景，扩展到 `Text` 或至少 `String(5000)`。

### 2.7 系统通知与聊天消息混杂
- **现状**：审核通知等系统消息也使用 Message 表，但与聊天消息在同一列表，无区分。
- **建议**：在聊天界面上方新增"通知"Tab，或者在消息列表中区分类型图标。

## 三、支付模块

### 3.1 不会自动生成账单
- **现状**：租客需要手动创建支付记录（选择合同、填写金额）。
- **建议**：合同生效后自动生成每月待支付账单，通过定时任务检查到期日。

### 3.2 逾期天数/滞纳金无自动计算
- **现状**：`overdue_days` 和 `overdue_fee` 字段存在，但从未被自动计算。
- **建议**：添加定时任务（例如每日运行），检查 `due_date` 逾期的账单，自动计算逾期天数和滞纳金。

### 3.3 无真实支付集成
- **现状**：点击"支付"直接调用 `updatePayment` 将状态改为 `paid`，无真实支付网关。
- **建议**：集成支付宝/微信支付的沙箱接口，或至少增加"标记为已支付"的权限控制（需要房东/admin确认）。

### 3.4 租客支付无权限校验
- **现状**：租客可以调用 `updatePayment` 将 `pending` 改为 `paid`，有安全风险。
- **建议**：`updatePayment` 只允许管理员或支付网关回调修改状态为 `paid`；租客只能发起支付请求。

### 3.5 房东端收款列表过于简单
- **现状**：`LandlordPayments.vue` 仅展示列表，无统计、无催缴功能。
- **建议**：添加应收总额、已收总额、逾期总额统计卡片；增加"催缴"按钮发送提醒消息。

## 四、预约看房模块

### 4.1 无冲突检测
- **现状**：同一房源在同一时间段可以被多个租客预约。
- **建议**：创建预约时检查该时间段该房源是否已有 `pending` / `approved` 状态的预约。

### 4.2 预约成功后无通知
- **现状**：租客预约后，房东不会收到任何通知。
- **建议**：创建预约后自动通过 Message 表发送通知给房东。

### 4.3 预约完成后状态不自动更新
- **现状**：预约时间过了之后，状态仍为 `approved` 或 `pending`。
- **建议**：添加定时任务或前端逻辑，将过期的预约自动标记为 `completed`。

### 4.4 预约时无房源状态校验
- **现状**：租客可以预约已出租（`rented`）的房源。
- **建议**：只有 `vacant` 状态的房源才允许被预约。

### 4.5 预约列表无房源标题/地址
- **现状**：`TenantBookings.vue` 只显示"房源 #{{ property_id }}"，不友好。
- **建议**：通过 API 联表查询或使用 useNameResolver 展示房源信息。

## 五、维修与投诉模块

### 5.1 前端未支持图片上传
- **现状**：`MaintenanceRequest` 和 `Complaint` 模型都有 `image_urls` 字段，但提交表单中没有图片上传。
- **建议**：添加图片上传组件，支持多图上传。

### 5.2 状态变更无通知
- **现状**：房东处理/解决维修或投诉后，租客不会收到通知。
- **建议**：状态变更时自动发送消息给相关人员。

### 5.3 租客无法确认维修完成
- **现状**：房东标记为 `resolved` 后，租客无法确认是否满意或重新开启。
- **建议**：租客可对 `resolved` 状态的维修申请确认关闭（`closed`）或重新开启。

### 5.4 投诉/维修列表无房源关联信息
- **现状**：`TenantMaintenance.vue` 和 `TenantComplaints.vue` 没有显示房源标题。
- **建议**：联表查询或使用 useNameResolver 展示房源名称。

## 六、前端通用问题

### 6.1 分页逻辑错误
- **现状**：几乎所有列表页的 `total` 都设为 `res.length`，而非后端的真实总数。
- **影响**：当数据超过一页时，分页控件无法正确显示总页数。
- **建议**：后端接口返回 `{ items: [...], total: N }` 而非直接返回数组；或添加 `X-Total-Count` 响应头。

### 6.2 占位图片使用外部链接
- **现状**：使用 `https://via.placeholder.com/...`，在国内网络可能无法访问。
- **建议**：使用本地占位图或 base64 内联图片。

### 6.3 缺少骨架屏加载
- **现状**：整页 `v-loading`，用户体验不够细腻。
- **建议**：卡片/表格使用 `el-skeleton` 骨架屏组件。

### 6.4 前端无搜索防抖
- **现状**：PropertyList 等搜索页面直接触发请求，无防抖。
- **建议**：使用 lodash 的 `debounce` 或自定义 hook。

### 6.5 前后端类型定义不一致
- **现状**：前端 JS 无类型安全，后端字段变更容易导致前端未及时更新。
- **建议**：可以考虑使用 TypeScript，或至少将 API 响应文档化。

### 6.6 登录后没有角色选择注册
- **现状**：注册页面没有角色字段（硬编码为 tenant），需要通过 DB 或 API 直接修改。
- **建议**：注册时让用户选择"租客"或"房东"角色。

## 七、后端架构与安全

### 7.1 CORS 配置过于宽松
- **现状**：`BACKEND_CORS_ORIGINS = ["*"]`，允许所有来源跨域。
- **建议**：生产环境限制为具体的前端域名。

### 7.2 SECRET_KEY 默认值未修改
- **现状**：`SECRET_KEY = "change-me-to-a-secure-random-string"`。
- **建议**：生产环境必须使用强随机密钥，通过 `.env` 配置。

### 7.3 无数据库迁移
- **现状**：使用 `Base.metadata.create_all()` 在启动时创建表。
- **建议**：使用 Alembic 管理数据库迁移，避免生产环境数据丢失。

### 7.4 无请求频率限制
- **现状**：API 无速率限制（rate limiting），容易被暴力破解或爬取。
- **建议**：使用 slowapi 或类似中间件添加请求限流。

### 7.5 日志系统不完善
- **现状**：只有 `AuditLog` 记录操作，无应用日志（request log、error log）。
- **建议**：配置结构化日志（如 loguru），统一记录请求和错误。

### 7.6 敏感信息未脱敏
- **现状**：审计日志等可能包含用户敏感信息。
- **建议**：审计日志中对身份证号、密码等字段脱敏。

### 7.7 空密码校验
- **现状**：注册时密码长度/复杂度无校验。
- **建议**：添加密码最小长度、复杂度要求。

### 7.8 Token 过期时间过长
- **现状**：`ACCESS_TOKEN_EXPIRE_MINUTES = 60 * 24 * 7`（7天）。
- **建议**：缩短过期时间（如1天），引入 refresh token 机制。

## 八、数据与模型问题

### 8.1 房源筛选维度少
- **现状**：只支持区域、户型过滤，缺少价格区间、朝向、类型等。
- **建议**：后端增加多个过滤参数，前端增加对应的过滤组件。

### 8.2 缺少房源收藏/关注功能
- **现状**：租客无法收藏喜欢的房源。
- **建议**：新增 `favorites` 表及对应 API、前端页面。

### 8.3 缺少房源评价/评分功能
- **现状**：合同到期后租客无法对房源/房东进行评价。
- **建议**：新增 `reviews` 表，合同结束后开放评价入口。

### 8.4 无浏览历史
- **现状**：用户无法查看自己浏览过的房源。
- **建议**：利用已有 `view_count` 机制，同时记录用户浏览历史。

### 8.5 验证/外键约束缺失
- **建议**：
  - Booking 的 `appointment_time` 应大于当前时间
  - Contract 的 `end_date` 应大于 `start_date`
  - Payment 的 `amount` 应大于 0

## 九、管理后台

### 9.1 管理员面板数据过于简单
- **现状**：Admin Dashboard 仅显示基础统计。
- **建议**：添加图表（ECharts）展示注册趋势、收入趋势、房源增长等。

### 9.2 缺少系统配置功能
- **建议**：增加系统参数配置（租金上限、押金倍数等）。

### 9.3 审计日志无法搜索
- **现状**：无过滤/搜索功能，日志多时难以定位。
- **建议**：增加按用户、操作类型、时间范围的过滤。

### 9.4 管理员无法查看/回复投诉
- **现状**：投诉处理在房东端，管理员无法介入。
- **建议**：Admin 应有投诉管理入口。

## 十、缺失的功能模块

### 10.1 实时消息通知
- **建议**：引入 WebSocket 或 Server-Sent Events，实现：
  - 新消息实时到达
  - 预约状态变更通知
  - 合同签署提醒
  - 支付成功/逾期提醒

### 10.2 智能搜索增强
- **现状**：`SmartSearch.vue` 未详审，但基础搜索功能有限。
- **建议**：实现全文搜索、拼音搜索、地图范围搜索。

### 10.3 多语言支持
- **现状**：界面文字全是中文硬编码。
- **建议**：使用 Vue I18n 实现国际化。

### 10.4 数据导出
- **现状**：无法导出任何数据。
- **建议**：导出房源、合同、支付记录到 CSV/Excel。

### 10.5 电子合同/PDF 生成
- **现状**：无合同文件生成功能。
- **建议**：使用 WeasyPrint 或 ReportLab 生成 PDF 格式合同。

### 10.6 忘记密码功能
- **现状**：没有 "忘记密码" 流程。
- **建议**：增加邮箱验证重置密码功能。

### 10.7 手机号/邮箱验证
- **现状**：注册无需验证邮箱或手机号。
- **建议**：增加邮箱/短信验证码校验。

## 十一、代码质量

### 11.1 重复的状态映射代码
- **现状**：每个前端页面都重复定义 `statusMap`、`statusTypeMap`、`statusLabel()`、`statusType()`。
- **建议**：抽取为全局 composable 或 utils，统一管理。

### 11.2 后端状态字符串硬编码
- **现状**：`status.in_([...])` 列表散落在各处。
- **建议**：定义全局常量或 Enum。

### 11.3 前端 API 调用无统一 loading 状态管理
- **现状**：每个页面独立管理 `loading`、`submitting` 等状态。
- **建议**：考虑使用 useRequest composable 统一管理请求状态。

### 11.4 部分后端接口直接在 endpoint 操作数据库
- **现状**：部分 endpoint 直接 `db.query(...)` 而非通过 CRUD 函数。
- **建议**：统一通过 `crud_xxx` 模块访问数据库。

### 11.5 配置文件硬编码数据库密码
- **现状**：`config.py` 中数据库密码直接写为 `"password"`，仅通过 .env 覆盖。
- **建议**：`config.py` 中不使用默认密码，强制要求设置环境变量。

## 十二、小问题 & 优化

### 12.1 Home.vue 最新房源排序
- **现状**：首页展示最新6套房源，但后端未指定排序，可能不是最新的。
- **建议**：后端 `get_properties` 按 `created_at` 降序排列。

### 12.2 PropertyList 显示已出租房源
- **现状**：列表会显示已出租（`rented`）房源。
- **建议**：前端默认隐藏已出租房源，或提供筛选条件。

### 12.3 房源详情页面浏览量未增加
- **现状**：`read_property` 接口未自增 `view_count`。
- **建议**：普通用户访问时自动 `view_count += 1`。

### 12.4 没有房源在地图上的展示
- **现状**：房源有 `latitude`/`longitude` 字段，但未使用。
- **建议**：在详情页用 Leaflet 或百度地图展示位置。

### 12.5 编辑合同接口权限过于宽松
- **现状**：`update_contract` 允许 landlord 修改 `monthly_rent`、`deposit` 等字段，即使合同已签署。
- **限制**：在双方签署前应允许修改，但修改后应重置签署状态。

### 12.6 合同与房源状态联动不完整
- **现状**：合同生效时 `property.status = rented`，终止时 `= vacant`。
- **问题**：如果合同到期（`expired`），房源状态不会自动恢复。
- **建议**：添加定时任务处理到期合同。

### 12.7 前端路由守卫不完善
- **现状**：admin 角色访问 `/tenant/` 和 `/landlord/` 会被拦截，但普通用户角色保护不够严格。
- **建议**：后端也应做角色权限校验（已有依赖注入，但部分接口缺失）。
